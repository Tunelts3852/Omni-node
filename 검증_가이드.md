## 3. Playwright/Node 기반 회귀

루트 `package.json`에는 의미 있는 `npm test`가 없고, 실제 검증은 개별 스크립트를 직접 실행하는 방식입니다.

먼저 의존성 준비:

```bash
npm ci
npx playwright install --with-deps chromium
```

대표 스크립트:

| 명령 | 의미 |
|---|---|
| `node omninode-dashboard/check-guard-threshold-lock.js` | guard threshold 잠금 검증 |
| `node omninode-dashboard/check-chat-multi-utils.js` | chat multi 유틸 계약 검증 |
| `node omninode-dashboard/check-guard-retry-timeline-browser-e2e.js` | retry timeline 브라우저 E2E |
| `node omninode-dashboard/check-ops-flow-performance.js` | 운영 패널 성능 점검 |
| `node omninode-middleware/check-p3-guard-smoke.js` | guard smoke |
| `node omninode-middleware/check-startup-probe-state.js` | startup probe 상태 검증 |
| `node omninode-middleware/check-guard-alert-dispatch.js` | guard alert dispatch 회귀 |

## 4. GitHub Actions에 반영된 대표 회귀

현재 루트 `.github/workflows/`에는 아래 워크플로가 있습니다.

| 워크플로 | 의미 |
|---|---|
| `guard-alert-dispatch-regression.yml` | dispatch mock/live 회귀 |
| `guard-retry-timeline-browser-e2e-regression.yml` | retry timeline 브라우저 E2E + p3/p7 묶음 회귀 |

공통 특징:

- Node.js 22 사용
- .NET 9 사용
- Gemini 키를 secure file 방식으로 강제
- 실행 결과를 artifact로 업로드

## 6. 라이브 회귀 전에 필요한 것

아래 조건이 없으면 일부 회귀는 실패하거나 의미가 줄어듭니다.

- Gemini API Key
- Chromium 설치
- `.NET 9 SDK`
- `sqlite3`
- 텔레그램 또는 외부 webhook 대상이 필요한 테스트의 경우 해당 자격정보

## 7. 헷갈리기 쉬운 점

- `npm test`는 현재 placeholder라서 검증 명령으로 보면 안 됩니다.
- 대시보드 앱은 브라우저 전역 객체를 쓰므로 실행 테스트와 단순 구문 점검을 구분해서 봐야 합니다.
- 검색 관련 라이브 회귀는 생성기 자체보다 `GeminiGroundedRetriever`, guard 상태 파일, 아티팩트 준비 상태 영향이 큽니다.
- 루틴 검증은 서버가 실제로 떠 있는 상태에서 보는 편이 훨씬 정확합니다.
