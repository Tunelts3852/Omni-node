# Omni-node 기술 스택 정리

업데이트 기준: 2026-03-07

## 1. 언어와 런타임

| 영역 | 스택 | 비고 |
|---|---|---|
| 코어 데몬 | C11 | UDS 소켓, 단일 인스턴스 락, 저수준 프로세스 제어 |
| 메인 서버 | C# / .NET 9 | `net9.0`, `PublishAot=true`, `System.Text.Json` source generation |
| 대시보드 | 정적 HTML + React 18 UMD + 순수 JS | 번들러 없이 서빙 |
| 샌드박스 | Python 3 | 제한된 별도 프로세스 실행 |
| 회귀 스크립트 | Node.js | Playwright, 각종 계약/회귀 검사 |

## 2. 프런트엔드 스택

`omninode-dashboard/index.html` 기준:

| 항목 | 사용 기술 |
|---|---|
| UI 라이브러리 | React 18 UMD, ReactDOM 18 UMD |
| 마크다운 렌더링 | `markdown-it`, `markdown-it-footnote` |
| HTML 정화 | `DOMPurify` |
| 수식 렌더링 | `KaTeX` |
| 폰트 | Plus Jakarta Sans, Noto Sans KR |
| 보조 처리 | Web Worker (`worker.js`) |

특징:

- `app.js` 단일 파일 중심 구조
- 현재 `app.js`는 9,170라인, `styles.css`는 4,332라인
- 루트 `package.json`은 앱 번들링이 아니라 Playwright 개발 의존성만 들고 있음

## 3. 백엔드/미들웨어 스택

주요 구성:

- `HttpListener` 기반 HTTP 서버
- `System.Net.WebSockets` 기반 WebSocket 게이트웨이
- `HttpClient` 기반 외부 API 호출
- `System.Text.Json` 기반 직렬화
- 파일 기반 상태 저장 + 원자적 쓰기
- SQLite3 CLI 호출 기반 메모리 인덱스 초기화

핵심 클래스:

| 파일 | 역할 |
|---|---|
| `Program.cs` | 전체 서비스 조립 |
| `WebSocketGateway.cs` | HTTP/WS, 인증, 대시보드 서빙 |
| `CommandService.*` | 대화/코딩/루틴/설정 중심 로직 |
| `LlmRouter.cs` | 제공자 호출 및 모델 라우팅 |
| `ProviderRegistry.cs` | 가용 제공자 계산 |
| `MemoryIndexSchemaBootstrap.cs` | SQLite FTS5 인덱스 부트스트랩 |

## 4. 외부 LLM 및 에이전트 연동

| 제공자/도구 | 방식 |
|---|---|
| Gemini | HTTP API |
| Groq | OpenAI 호환 HTTP API |
| Cerebras | HTTP API |
| GitHub Copilot | CLI 래퍼 (`gh`, `copilot`) |
| Codex | CLI 래퍼 (`codex`) |

자동 제공자 우선순위:

```text
gemini -> groq -> cerebras -> copilot -> codex
```

Codex 관련 현재 구현 포인트:

- Codex CLI 설치 여부 검사
- `codex login status`
- device auth 로그인 시작
- API Key 또는 CLI 로그인 상태를 동시에 고려

## 5. 검색/RAG/보안 스택

현재 검색 경로의 중심은 Gemini grounding입니다.

| 구성 | 역할 |
|---|---|
| `GeminiGroundedRetriever.cs` | `gemini-3.1-flash-lite-preview` + `google_search` 근거 수집 |
| `LegacyGeminiGroundingSearchGateway.cs` | 검색 재시도, count-lock, 종료 이유 관리 |
| `SearchEvidencePackBuilder.cs` | 근거 묶음 구성 |
| `SearchAnswerGuard.cs` | freshness/credibility/coverage fail-closed 평가 |
| `ExternalContentGuard.cs` | 외부 콘텐츠를 `EXTERNAL_UNTRUSTED_CONTENT`로 감싸 프롬프트 경계 생성 |

핵심 정책:

- 검색 근거가 부족하면 답변을 차단
- 최신성/신뢰성 검증 실패 시 확정형 응답 금지
- 외부 웹 콘텐츠는 신뢰된 내부 지시문으로 취급하지 않음

## 6. 상태 저장과 보안

| 영역 | 방식 |
|---|---|
| 대화 저장 | JSON 파일 |
| 메모리 노트 | 파일 트리 |
| 메모리 인덱스 | SQLite + FTS5 |
| 인증 세션 | 파일 + 서명 토큰 |
| 감사 로그 | JSONL 파일 |
| 시크릿 | macOS Keychain 또는 로컬 secure store 또는 `*_FILE` |

시크릿 로딩 우선순위:

1. macOS Keychain
2. 로컬 secure store (`~/.config/omninode/secrets.json`)
3. `*_FILE`
4. 직접 환경변수

직접 환경변수는 코드상 deprecated 경고를 남깁니다.

## 7. 코딩 실행 스택

`UniversalCodeRunner`가 기본 실행을 맡고, Python은 `omninode-sandbox/executor.py`로 분리됩니다.

지원 언어:

- Python
- JavaScript
- Bash
- C
- C++
- C#
- Java
- Kotlin
- HTML
- CSS

특징:

- HTML/CSS는 실행 대신 파일 저장
- Python은 timeout, memory, CPU limit 적용 가능
- 각 실행은 날짜별 run directory에 저장

## 8. 테스트/검증 스택

| 영역 | 도구 |
|---|---|
| 프런트 구문 점검 | `node --check` |
| 브라우저 E2E | Playwright |
| 백엔드 smoke/regression | Node 스크립트 |
| CI | GitHub Actions |

루트 `package.json` 기준 Playwright 버전:

- `playwright`: `^1.58.2`

## 9. 배포 스택

| 파일 | 역할 |
|---|---|
| `deploy/linux/omninode.service` | systemd 서비스 템플릿 |
| `deploy/macos/com.omninode.agent.plist` | launchd plist 템플릿 |

둘 다 `/opt/omninode/bin/omninode_middleware` 실행을 전제로 잡혀 있습니다.

## 10. 현재 코드 구조 특성

이 저장소는 “잘게 분리된 모듈형 백엔드 + SPA 빌드 파이프라인” 구조와는 다릅니다.

관찰 포인트:

- `CommandService.Commands.cs`가 14,206라인으로 매우 큼
- `WebSocketGateway.cs`도 7,158라인 규모
- 대시보드는 번들러/타입체커 없이 큰 단일 파일로 운영 중
- 제품 소스와 실험/산출물 디렉터리가 한 저장소에 공존

즉, 기술 스택은 다양하지만 코드 조직은 강한 중앙집중형 모놀리식 성격을 갖습니다.
