# Omni-node 아키텍처 흐름

업데이트 기준: 2026-03-07

이 문서는 저장소를 “어떤 파일이 있느냐”가 아니라 “요청이 들어왔을 때 실제로 어떻게 흐르느냐” 기준으로 정리합니다.

## 1. 전체 런타임 흐름

```mermaid
flowchart TD
    WEB["웹 대시보드"] --> G["WebSocketGateway"]
    TG["텔레그램"] --> TL["TelegramUpdateLoop"]
    G --> CS["CommandService"]
    TL --> CS
    CS --> MEM["대화/메모리 계층"]
    CS --> SEARCH["검색/근거 계층"]
    CS --> CODE["코드 실행 계층"]
    CS --> UDS["UdsCoreClient"]
    CS --> REG["ProviderRegistry / ToolRegistry"]
    REG --> ROUTER["LlmRouter"]
    ROUTER --> PROVIDERS["Gemini / Groq / Cerebras / Copilot / Codex"]
    UDS --> CORE["omninode-core"]
    CODE --> SANDBOX["omninode-sandbox"]
```

핵심은 `CommandService`가 거의 모든 기능의 허브라는 점입니다.

## 2. 시작 시퀀스

`Program.cs` 기준 부팅 순서는 아래와 같습니다.

1. 환경변수와 시크릿 로드
2. `RuntimeSettings`, `TelegramClient`, `SessionManager`, `UdsCoreClient` 생성
3. `LlmRouter`, `GroqModelCatalog`, `CodexCliWrapper`, `CopilotCliWrapper` 준비
4. 대화 저장소, 메모리 저장소, 코드 실행기, 툴 레지스트리 구성
5. 메모리 인덱스 bootstrap + 1회 sync
6. `CommandService` 생성
7. `WebSocketGateway`와 `TelegramUpdateLoop` 시작

```mermaid
flowchart LR
    A["AppConfig.LoadFromEnvironment"] --> B["RuntimeSettings / SecretLoader"]
    B --> C["MemoryIndexSchemaBootstrap"]
    C --> D["CommandService"]
    D --> E["WebSocketGateway"]
    D --> F["TelegramUpdateLoop"]
```

## 3. 인증 흐름

대시보드는 WebSocket 세션과 OTP를 사용합니다.

```mermaid
sequenceDiagram
    participant U as 사용자 브라우저
    participant G as WebSocketGateway
    participant S as SessionManager
    participant T as TelegramClient

    U->>G: WebSocket 연결
    G->>S: pending 세션 생성
    U->>G: request_otp
    G->>S: OTP 조회
    G->>T: SendOtpAsync
    T-->>G: 성공 또는 실패
    G-->>U: otp_request_result
    U->>G: auth(otp)
    G->>S: Authenticate
    S-->>G: trusted auth token 발급
    G-->>U: auth_result
```

보조 포인트:

- 텔레그램 미설정 시 로컬 OTP fallback 가능
- trusted auth token은 서명되고 일 단위 키 로테이션을 사용
- 브라우저는 이후 `resume_auth`로 세션 복구 가능

## 4. 일반 대화 흐름

```mermaid
flowchart TD
    A["대화 요청"] --> B["WebSocketGateway 메시지 수신"]
    B --> C["CommandService.ExecuteAsync"]
    C --> D["입력 준비 / 메모리 검색 / 대화 스레드 조회"]
    D --> E["ProviderRegistry로 제공자 선택"]
    E --> F["LlmRouter 호출"]
    F --> G["응답 후 대화 저장"]
    G --> H["메모리 노트/요약 갱신"]
    H --> I["WebSocket으로 결과 전송"]
```

자동 제공자 선택 시 우선순위는 다음과 같습니다.

```text
gemini -> groq -> cerebras -> copilot -> codex
```

## 5. 최신정보/검색 응답 흐름

현재 최신정보 경로는 Gemini grounding 단일 검색 파이프라인을 중심으로 돌아갑니다.

```mermaid
flowchart TD
    Q["시간 민감 질의"] --> SG["LegacyGeminiGroundingSearchGateway"]
    SG --> GR["GeminiGroundedRetriever"]
    GR --> EP["SearchEvidencePackBuilder"]
    EP --> GUARD["SearchAnswerGuard"]
    GUARD -->|통과| GEN["최종 생성기(Gemini/Groq/Cerebras/Copilot/Codex)"]
    GUARD -->|차단| BLOCK["fail-closed 응답"]
    GEN --> OUT["인용 포함 응답"]
```

세부 규칙:

- `GeminiGroundedRetriever`는 `google_search` 툴을 붙여 근거를 수집
- 외부 웹 본문은 `EXTERNAL_UNTRUSTED_CONTENT` 경계로 감쌈
- `SearchAnswerGuard`는 `coverage`, `freshness`, `credibility`를 평가
- count-lock 미충족이면 최종 답변 대신 차단 가능

## 6. 코딩 실행 흐름

```mermaid
flowchart TD
    A["코딩 요청"] --> B["CommandService"]
    B --> C["언어/모드 판정"]
    C --> D["LLM으로 코드 생성 또는 복구"]
    D --> E["UniversalCodeRunner"]
    E -->|Python| F["PythonSandboxClient -> executor.py"]
    E -->|그 외| G["로컬 shell 실행 또는 파일 저장"]
    F --> H["실행 결과 수집"]
    G --> H
    H --> I["변경 파일/요약/실행 로그 반환"]
```

특징:

- HTML/CSS는 실행 대신 파일 저장
- Python은 별도 샌드박스 경로 사용
- 실행 산출물은 날짜 기준 디렉터리에 쌓임

## 7. 루틴/스케줄러 흐름

루틴은 대시보드와 텔레그램 양쪽에서 관리할 수 있습니다.

```mermaid
flowchart TD
    UI["웹/텔레그램에서 루틴 생성"] --> CFG["CommandService.Config"]
    CFG --> STATE["~/.omninode/routines.json 저장"]
    STATE --> SCH["RoutineSchedulerLoop"]
    SCH --> RUN["RunRoutineNowAsync"]
    RUN --> EXEC["코드 실행 또는 브라우저 에이전트 실행"]
    EXEC --> OUT["coding/routines/<id>/runs/... 산출물"]
    OUT --> NOTE["최근 실행 상태/출력 갱신"]
```

현재 구조상 루틴은 두 층을 함께 사용합니다.

- 상태 저장: `~/.omninode/routines.json`
- 작업 산출물: `coding/routines/`

## 8. 메모리/대화 저장 흐름

```mermaid
flowchart LR
    CHAT["대화/코딩 결과"] --> STORE["ConversationStore / MemoryNoteStore"]
    STORE --> FILES["JSON + markdown 파일"]
    FILES --> SYNC["MemoryIndexDocumentSync"]
    SYNC --> SQLITE["SQLite memory-index/main.sqlite"]
    SQLITE --> SEARCH["MemorySearchTool / MemoryGetTool"]
```

즉, 메모리는 단순 파일 저장과 검색용 인덱스를 동시에 유지하는 구조입니다.

## 9. 운영 관점에서 꼭 알아둘 흐름

- 프런트엔드는 정적 파일이라 미들웨어만 뜨면 같이 열립니다.
- 텔레그램 미설정 상태에서도 웹 대시보드는 단독으로 테스트 가능합니다.
- 검색 품질 문제는 보통 생성기보다 `Retriever -> Evidence Pack -> Guard` 구간에서 추적해야 합니다.
- 루틴은 단순 cron 문자열 저장이 아니라 실행 코드, 실행 이력, 재시도 정책, 알림 정책까지 함께 관리합니다.
