# Omni-node 기술 스택 정리

업데이트 기준: 2026-03-13

## 1. 언어와 런타임

| 영역 | 스택 | 비고 |
|---|---|---|
| 코어 데몬 | C11 | UDS 소켓, 단일 인스턴스 락, 저수준 프로세스 제어 |
| 메인 서버 | C# / .NET 9 | `net9.0`, `PublishAot=true`, `System.Text.Json` source generation |
| 대시보드 | 정적 HTML + React 18 UMD + ES module JS | 번들러 없이 서빙 |
| 샌드박스 | Python 3 | 제한된 별도 프로세스 실행 |
| 회귀 스크립트 | Node.js | Playwright, 각종 계약/회귀 검사 |

## 2. 프런트엔드 스택

`apps/omninode-dashboard/index.html` 기준:

| 항목 | 사용 기술 |
|---|---|
| UI 라이브러리 | React 18 UMD, ReactDOM 18 UMD |
| 마크다운 렌더링 | `markdown-it`, `markdown-it-footnote` |
| HTML 정화 | `DOMPurify` |
| 수식 렌더링 | `KaTeX` |
| 폰트 | Plus Jakarta Sans, Noto Sans KR |
| 보조 처리 | Web Worker (`worker.js`) |

특징:

- `app.js`는 상태 조립 엔트리이고, 실제 렌더/상태/WS 보조 로직은 `modules/` 아래 ES module로 분리됨
- 로직 탭은 `dashboard-logic-renderers.js`, `logic-state.js`, `ws-logic.js`로 캔버스/UI/WS 요청을 분리하고, `app.js`는 상태 조립과 제스처 연결을 맡음
- 현재 `app.js`는 6,610라인, `styles.css`는 7,480라인
- 루트 `package.json`의 `npm test`는 `scripts/run-omninode-tests.mjs`를 실행하는 로컬 smoke 엔트리다

## 3. 백엔드/미들웨어 스택

주요 구성:

- `HttpListener` 기반 HTTP 서버
- `System.Net.WebSockets` 기반 WebSocket 게이트웨이
- `HttpClient` 기반 외부 API 호출
- `System.Text.Json` 기반 직렬화
- 파일 기반 상태 저장 + 원자적 쓰기
- SQLite3 CLI 호출 기반 메모리 인덱스 초기화

핵심 클래스:

| 파일 | 역할 |
|---|---|
| `Program.cs` | 전체 서비스 조립 |
| `CoreProcessBootstrapper.cs` | 코어 소켓 응답 확인과 `omninode_core` 자동 부트스트랩 |
| `WebSocketGateway.cs` | WebSocket 수명주기와 dispatcher 조립 |
| `AuthSessionGateway.cs` | OTP 요청, 인증, trusted auth 복구 |
| `Ws*CommandDispatcher.cs` | WS 명령을 기능별로 분기 |
| `WsLogicCommandDispatcher.cs` | 로직 그래프 전용 WS 명령 분기 |
| `WsDoctorCommandDispatcher.cs` / `WsContextCommandDispatcher.cs` / `WsPlanningCommandDispatcher.cs` / `WsTaskCommandDispatcher.cs` / `WsNotebookCommandDispatcher.cs` | 설정 탭의 doctor, 프로젝트 문맥, 계획, task graph, notebook 명령 분기 |
| `CommandService.*` | 대화/코딩/루틴/설정 중심 partial 서비스 |
| `Application/Logic/*` | `logic.graph.v1` 모델, source-generated JSON, 런타임 코디네이터 |
| `CommandService.LogicPathBrowser.cs` | 로직 탭 경로 브라우저와 허용 루트 탐색 |
| `LlmRouter.cs` | 제공자 호출 및 모델 라우팅 |
| `Application/ApplicationServiceContracts.cs` | 애플리케이션 서비스 계약 |
| `Infrastructure/Persistence/PersistenceContracts.cs` | 대화/메모리/루틴/산출물 저장 계약 |
| `Infrastructure/Paths/StatePathResolver.cs` | 상태 파일 경로 canonical 해석 |
| `ProviderRegistry.cs` | 가용 제공자 계산 |
| `MemoryIndexSchemaBootstrap.cs` | SQLite FTS5 인덱스 부트스트랩 |

로직 그래프 구현 포인트:

- 저장 포맷은 `logic.graph.v1`
- 저장 모델은 일반 루틴 저장소를 확장한 `ExecutionMode=logic_graph`
- 실행기는 `LogicGraphRuntimeCoordinator`와 `CommandService.LogicGraphs.cs`가 맡고, 결과 직렬화는 `LogicGraphJsonContext.cs` source generation을 사용한다
- 경로 입력이 필요한 노드는 `logic_path_list`와 `CommandService.LogicPathBrowser.cs`를 통해 파일/메모리 루트를 탐색한다

## 4. 외부 LLM 및 에이전트 연동

| 제공자/도구 | 방식 |
|---|---|
| Gemini | HTTP API |
| Groq | OpenAI 호환 HTTP API |
| Cerebras | HTTP API |
| GitHub Copilot | CLI 래퍼 (`gh`, `copilot`) |
| Codex | CLI 래퍼 (`codex`) |

자동 제공자 선택은 이제 전역 고정 우선순위가 아니라 category chain + fallback 구조입니다.

```text
generalChat: gemini -> groq -> cerebras -> copilot -> codex
planner: gemini -> groq -> cerebras -> codex -> copilot
reviewer: codex -> gemini -> groq -> cerebras -> copilot
deepCode: codex -> copilot -> gemini -> groq -> cerebras
quickFix: groq -> cerebras -> gemini -> copilot -> codex
visualUi: gemini -> codex -> groq -> cerebras -> copilot
```

중요 사항:

- 기존 `gemini -> groq -> cerebras -> copilot -> codex`는 이제 category 내부 fallback 체인의 한 예시로만 남습니다.
- planner / reviewer / coding / search 판단은 `~/.omninode/routing-policy.json` override를 통해 바꿀 수 있습니다.
- `ProviderRegistry`는 availability뿐 아니라 visual / code / search / cli-auth-required / background-safe capability도 함께 계산합니다.

Codex 관련 현재 구현 포인트:

- Codex CLI 설치 여부 검사
- `codex login status`
- device auth 로그인 시작
- API Key 또는 CLI 로그인 상태를 동시에 고려

## 5. 검색/RAG/보안 스택

현재 검색 문서의 기준 추상화는 특정 gateway가 아니라, Gemini grounding과 fallback composer/local cache fallback을 함께 묶는 검색 파이프라인입니다.

| 구성 | 역할 |
|---|---|
| `GeminiGroundedRetriever.cs` | 검색 파이프라인의 primary retriever. `gemini-3.1-flash-lite-preview` + `google_search` 근거 수집 |
| `LegacyGeminiGroundingSearchGateway.cs` | 검색 파이프라인 orchestration. 재시도, count-lock, 종료 이유 관리 |
| `SearchPipelineContracts.cs` | 검색 파이프라인 계약 (`ISearchRetriever`, `IEvidencePackBuilder`, `ISearchGuard`, `ISearchAnswerComposer`) |
| `SearchEvidencePackBuilder.cs` | 근거 묶음 구성 |
| `SearchAnswerGuard.cs` | freshness/credibility/coverage fail-closed 평가 |
| `DefaultSearchAnswerComposer.cs` | grounded 실패 또는 저하 시 evidence/cache fallback 응답 구성 |
| `ExternalContentGuard.cs` | 외부 콘텐츠를 `EXTERNAL_UNTRUSTED_CONTENT`로 감싸 프롬프트 경계 생성 |

핵심 정책:

- grounding 실패 시에도 검색 파이프라인의 `ISearchAnswerComposer` 단계가 근거를 다시 조합해 안전한 fallback 응답을 만들 수 있음
- `SearchRetrieverPath.LocalCacheFallback`은 검색 파이프라인의 대체 retriever 경로로 최근 검증 스냅샷 기반 응답을 허용함
- 검색 근거가 부족하면 답변을 차단
- 최신성/신뢰성 검증 실패 시 확정형 응답 금지
- 외부 웹 콘텐츠는 신뢰된 내부 지시문으로 취급하지 않음

## 6. 상태 저장과 보안

| 영역 | 방식 |
|---|---|
| 대화 저장 | JSON 파일 |
| 메모리 노트 | 파일 트리 |
| 메모리 인덱스 | SQLite + FTS5 |
| 인증 세션 | 파일 + 서명 토큰 |
| 감사 로그 | JSONL 파일 |
| 시크릿 | macOS Keychain 또는 로컬 secure store 또는 `*_FILE` |

추가 상태 메모:

- `~/.omninode/routines.json`은 일반 루틴과 로직 그래프 정의를 함께 저장한다
- 로직 그래프 실행 산출물은 `workspace/.runtime/logic/<graph-id>/<run-id>/` 아래에 `graph.json`, `snapshot.json`, `events.log`로 남는다

시크릿 로딩 우선순위:

1. macOS Keychain
2. 로컬 secure store (`~/.config/omninode/secrets.json`)
3. `*_FILE`
4. 직접 환경변수

직접 환경변수는 코드상 deprecated 경고를 남깁니다.

## 7. 코딩 실행 스택

`UniversalCodeRunner`가 기본 실행을 맡고, Python은 `apps/omninode-sandbox/executor.py`로 분리됩니다.

지원 언어:

- Python
- JavaScript
- Bash
- C
- C++
- C#
- Java
- Kotlin
- HTML
- CSS

특징:

- HTML/CSS는 실행 대신 파일 저장
- Python은 timeout, memory, CPU limit 적용 가능
- 코딩 탭 `단일 / 오케스트레이션 / 다중`은 모두 `workspace/coding/runs/<timestamp>-<mode>-<suffix>/` 아래 실행별 작업 폴더를 공유
- 다중 코딩은 같은 실행 폴더 아래 `<provider>-<model>/` 하위 폴더로 결과를 분리
- 요청에 출력값이 있으면 stdout까지 함께 검증하고, 생성/수정 파일 존재와 파일 내용도 같이 확인함
- 코딩 프로필은 provider/model별 설정 위에 언어별 규칙을 다시 얹음
- `java`, `c`는 실제 컴파일 후 실행으로 검증하고, `html`은 Playwright 브라우저 검증을 사용함
- Copilot 코딩 경로는 현재 `gpt-5-mini`로 고정하고, 다중 파일 복구는 direct recovery와 deterministic repair를 함께 사용함

## 10. 현재 코드 구조 특성

이 저장소는 여전히 단일 프로세스 중심이지만, 이전보다 기능 경계를 더 분명히 가진 모듈형 모놀리스에 가깝습니다.

관찰 포인트:

- `CommandService.Commands.cs`는 84라인 facade로 유지되고, 본문은 기능별 partial 파일로 분리됨
- `WebSocketGateway.cs`는 4,545라인이며 인증/정적 파일/API/WS 명령이 별도 gateway/dispatcher로 추출됨
- `CommandService.Utils.cs`는 6,059라인이며 코딩/경로/실행 보조와 일부 공용 유틸을 여전히 많이 담고 있음
- `CommandService.LogicGraphs.cs`는 2,908라인이며 로직 그래프 저장, DAG 검증, 직접 실행을 담당함
- 대시보드는 여전히 번들러/타입체커 없이 운영되지만 renderer/state/ws 로직은 `modules/`로 분해됨
- 제품 소스와 실험/산출물 디렉터리가 한 저장소에 공존

즉, 기술 스택은 다양하고 배포 단위는 단순하지만, 내부 구조는 중앙 허브 의존을 줄이는 방향으로 재정리된 상태입니다.
