# Omni-node 사용법 빠른 시작

업데이트 기준: 2026-03-13

이 문서는 저장소를 처음 다시 열었을 때 가장 빠르게 부팅하고, 대시보드에 접속하고, 핵심 기능을 확인하는 절차만 정리합니다.

## 1. 사전 준비

최소 권장 도구:

| 항목 | 용도 |
|---|---|
| `cc` | `omninode-core` 빌드 |
| `.NET SDK 9` | `omninode-middleware` 빌드/실행 |
| `python3` | 샌드박스 실행기, 코딩 워크플로 |
| `sqlite3` | 메모리 인덱스 초기화 |
| `node` + `npm` | Playwright 및 회귀 스크립트 |
| `gh` + `copilot` | Copilot CLI 연동 시 |
| `codex` | Codex CLI 연동 시 |

선택 항목:

- Telegram Bot Token / Chat ID
- Gemini / Groq / Cerebras API Key
- Codex API Key 또는 Codex CLI 로그인

코딩탭 개발 명령은 가능한 경우 `node`/`python3` 런타임과 `vite`/`uvicorn` 같은 npm/pip CLI 패키지를 자동 설치 재시도합니다. 기준은 macOS `brew`, Linux `apt-get`이며, 네트워크나 sudo 무인 실행이 막힌 환경에서는 여전히 실패할 수 있습니다.

## 2. 가장 빠른 부팅 순서

### 2.0 원커맨드 부팅

전역 명령이 등록돼 있으면 아래 세 개로 끝난다.

```bash
Omni-node setup
Omni-node
Omni-node shutdown
```

- `Omni-node setup`: macOS Homebrew 기준 의존성 설치, `npm ci`, 코어 빌드, .NET 빌드, 샌드박스 smoke, `npm test`, 전역 명령 등록까지 한 번에 수행
- `Omni-node`: 서버 시작 후 `http://127.0.0.1:8080/` 준비 상태까지 기다린다
- `Omni-node shutdown`: CLI가 관리하는 미들웨어와 해당 실행에서 새로 띄운 코어를 종료한다

처음 한 번은 저장소 루트에서 아래처럼 직접 실행해 전역 명령을 등록할 수 있다.

```bash
./scripts/Omni-node setup
```

CLI 실행 상태는 `~/.omninode/cli/` 아래에 기록된다.

### 2.1 코어 실행

```bash
make -C apps/omninode-core
./apps/omninode-core/omninode_core
```

코어는 `/tmp/omninode_core.<uid>.sock` UDS 소켓과 `/tmp/omninode.<uid>.lock` 락 파일을 사용합니다.

### 2.2 미들웨어 실행

다른 터미널에서:

```bash
dotnet run --project apps/omninode-middleware/OmniNode.Middleware.csproj
```

기본 포트는 `8080`입니다.

참고:

- 코어를 수동으로 먼저 띄우는 것이 가장 명확하지만, 미들웨어도 시작 시 `core_socket` 응답이 없으면 `apps/omninode-core/omninode_core`를 자동으로 띄우려 시도합니다.
- stale socket/lock만 남아 있는 상태에서도 같은 자동 복구 경로를 먼저 탑니다.

환경 진단만 먼저 확인하려면:

```bash
dotnet run --project apps/omninode-middleware/OmniNode.Middleware.csproj -- doctor
```

### 2.3 대시보드 접속

- 대시보드: `http://127.0.0.1:8080/`
- 헬스체크: `http://127.0.0.1:8080/healthz`
- 준비상태: `http://127.0.0.1:8080/readyz`

별칭:

- `http://127.0.0.1:8080/health`
- `http://127.0.0.1:8080/ready`

`readyz`는 첫 WebSocket roundtrip이 잡혀야 200이 됩니다. 현재 대시보드는 접속 직후 초기 `ping`을 자동 전송하므로, 페이지가 정상 연결되면 별도 수동 `ping` 없이 준비상태가 올라갑니다.

## 3. 인증 방식

대시보드는 WebSocket OTP 인증을 사용합니다.

기본 흐름:

1. 브라우저에서 대시보드 접속
2. `OTP 요청` 실행
3. 텔레그램이 설정되어 있으면 텔레그램으로 OTP 수신
4. 텔레그램이 없으면 콘솔에 로컬 fallback OTP가 출력됨
5. OTP 입력 후 세션 인증 완료

기본적으로 `OMNINODE_ENABLE_LOCAL_OTP_FALLBACK=true`라서 텔레그램 미설정 상태에서도 로컬 테스트가 가능합니다.

## 4. 추천 환경변수 예시

시크릿은 직접 환경변수보다 `*_FILE` 방식이 안전합니다. 파일 권한은 `0600` 이상으로 맞추는 것이 전제입니다.

예시 시크릿 디렉터리는 `~/.omninode/keys`로 통일합니다.

```bash
mkdir -p "$HOME/.omninode/keys"

printf 'YOUR_GEMINI_KEY' > "$HOME/.omninode/keys/gemini_api_key"
printf 'YOUR_GROQ_KEY' > "$HOME/.omninode/keys/groq_api_key"
printf 'YOUR_CEREBRAS_KEY' > "$HOME/.omninode/keys/cerebras_api_key"
chmod 600 "$HOME/.omninode/keys/"*

export OMNINODE_GEMINI_API_KEY_FILE="$HOME/.omninode/keys/gemini_api_key"
export OMNINODE_GROQ_API_KEY_FILE="$HOME/.omninode/keys/groq_api_key"
export OMNINODE_CEREBRAS_API_KEY_FILE="$HOME/.omninode/keys/cerebras_api_key"
export OMNINODE_WORKSPACE_ROOT="/path/to/Omni-node/workspace/coding"
export OMNINODE_GATEWAY_STARTUP_PROBE=1
```

이 문서의 워크스페이스 예시는 항상 canonical 경로인 `/path/to/Omni-node/workspace/coding`을 기준으로 합니다. 루트 `coding`은 하위 호환 alias입니다.

Codex를 API Key로 쓸 경우:

```bash
printf 'YOUR_CODEX_KEY' > "$HOME/.omninode/keys/codex_api_key"
chmod 600 "$HOME/.omninode/keys/codex_api_key"
export OMNINODE_CODEX_API_KEY_FILE="$HOME/.omninode/keys/codex_api_key"
```

텔레그램을 함께 붙일 경우:

```bash
printf 'YOUR_TELEGRAM_BOT_TOKEN' > "$HOME/.omninode/keys/telegram_bot_token"
printf 'YOUR_TELEGRAM_CHAT_ID' > "$HOME/.omninode/keys/telegram_chat_id"
chmod 600 "$HOME/.omninode/keys/telegram_"*

export OMNINODE_TELEGRAM_BOT_TOKEN_FILE="$HOME/.omninode/keys/telegram_bot_token"
export OMNINODE_TELEGRAM_CHAT_ID_FILE="$HOME/.omninode/keys/telegram_chat_id"
export OMNINODE_TELEGRAM_ALLOWED_USER_ID="YOUR_TELEGRAM_USER_ID"
```

## 5. 대시보드 사용 순서

현재 전역 탭은 `대화`, `루틴`, `로직`, `코딩`, `설정` 순서로 보인다.

### 5.1 대화 탭

- 단일 / 오케스트레이션 / 다중 대화 모드 지원
- 오케스트레이션은 입력 성격에 따라 워커 역할을 자동으로 나눠서 초안, 리스크 점검, 반례·예외, 실행안 구체화, 최종 누락 점검 같은 관점을 분담한 뒤 통합함
- 다중 LLM은 모델별 답변을 캐러셀로 하나씩 넘겨 보며, 바로 아래 다음 메시지에서 `공통 요약 / 공통 핵심 / 부분 차이`를 따로 확인하는 방식임
- 메모리 노트 연결, URL 컨텍스트, 웹 검색 on/off 사용 가능
- 링크만 단독으로 붙여 넣거나 `이거 봐봐`, `이 링크 설명해줘`처럼 짧은 말만 붙여도 단일 대화와 텔레그램 단일 대화에서는 URL 참조 경로를 우선 사용해 해당 페이지/문서/저장소 내용을 설명함
- GitHub 저장소 루트 링크는 가능하면 공개 README를 직접 읽어 근거로 사용하므로, `맥 미니 관련 내용만`, `Groq 얘기만` 같은 질문도 저장소 본문 기준으로 좁혀 답함
- 최신정보 질의는 Gemini grounding 우선 경로와 `SearchAnswerGuard`, evidence/cache fallback 정책 영향을 받음

### 5.2 코딩 탭

- 단일 / 오케스트레이션 / 다중 코딩 모드 지원
- `단일 코딩`: 선택한 모델 1개가 처음부터 끝까지 구현, 실행, 검증, 수정까지 직접 수행함
- `오케스트레이션 코딩`: 입력과 모델 구성에 따라 `기획 -> 개발 -> 검증 및 테스트 -> 수정` 단계를 자동 분담하고, 각 단계가 같은 실행 폴더를 이어서 사용함
- `오케스트레이션 코딩`을 입력 없이 실행하면 워커들이 역할을 스스로 협의해 실행 가능한 개선 작업 1건을 고르는 자동 모드로 시작할 수 있음
- `다중 코딩`: 선택한 각 provider/model이 서로 독립된 하위 폴더에서 처음부터 끝까지 완주하고, 대화와 `최근 코딩 결과` 카드에서 모델 캐러셀 + `공통 요약 / 공통점 / 차이 / 추천` 정리로 비교함
- 세 모드 모두 같은 코딩 루트 `workspace/coding/runs/`를 쓰며, 실행할 때마다 `workspace/coding/runs/<timestamp>-<mode>-<suffix>/` 폴더를 새로 만듦
- 다중 코딩의 모델별 산출물은 같은 실행 폴더 아래 `<provider>-<model>/` 하위 폴더에 따로 저장됨
- 코딩 프로필은 provider/model별 설정 위에 언어별 규칙을 한 번 더 얹는다. 현재 기본 실검 기준 언어는 `python`, `javascript`, `java`, `c`, `html`이다
- 기본 코딩 모델은 Codex `gpt-5.4`, Copilot `gpt-5-mini`, Gemini `gemini-3-flash-preview`, Groq `meta-llama/llama-4-scout-17b-16e-instruct`, Cerebras `gpt-oss-120b`다
- Cerebras의 `zai-glm-4.7 (preview)`와 `qwen-3-235b-a22b-instruct-2507 (preview)`는 공식 문서 기준으로 수요가 높아 free-tier rate limit이 일시적으로 줄어들 수 있으므로, 429가 나면 잠시 후 재시도하거나 `gpt-oss-120b`를 사용한다
- Copilot 코딩 경로는 현재 `gpt-5-mini`로 고정해서 사용한다
- 코딩 루프는 모델이 준 파일 경로와 실행 명령의 개행/탭을 정규화하고, 요청에 파일명이 명시돼 있으면 그 경로를 우선 사용함
- 절대경로, 스마트 따옴표로 감싼 경로, 예전 `workspace/coding/...` 경로를 넣어도 현재 실행 폴더 기준 상대 경로로 정리해 중첩 경로 오류를 막음
- 최종 검증은 요청에 출력값이 명시된 경우 stdout까지 확인하고, 실패하면 같은 실행 폴더에서 1회 추가 수정 패스를 시도함
- 최종 검증은 생성/수정된 파일 존재 여부도 함께 확인하며, 요약/비교 카드에도 stdout 요약과 변경 파일 정보가 같이 반영됨
- `java`, `c`는 실제 컴파일 후 실행으로 검증하고, `html`은 브라우저 렌더링 텍스트와 selector/style까지 확인하는 경로를 사용함
- `파일 하나` 요청에서 생긴 기본 엔트리 파일(`main.py` 등) 잔여는 최종 성공 시 정리함
- Copilot의 단순 `파일 하나 + 문자열 출력` smoke 요청은 원본 사용자 입력 기준 결정론적 fast-path로 바로 생성/검증할 수 있음
- `최근 코딩 결과` 카드와 오버레이에서 생성 파일 프리뷰, 실행 명령, stdout/stderr, 브라우저 iframe 프리뷰를 다시 확인할 수 있음
- `실행` 버튼은 마지막 검증 명령이 아니라 엔트리 파일 실행을 우선 사용하며, 입력형 CLI는 `stdin`을 넣어 다시 실행할 수 있음
- Python GUI/게임류는 최근 코딩 결과에서 재실행 시 사용자가 직접 닫을 때까지 살아 있도록 별도 실행 경로를 사용함
- 코딩탭 워커는 이제 제공자/모델별 실행 프로필을 사용하며, Codex/Copilot 계열은 경량 프롬프트·직생성 복구를 먼저 시도하고, Gemini/Groq/Cerebras 계열은 반복 루프 길이와 스냅샷 크기를 다르게 조절함
- 실행 이력은 기본적으로 `~/.omninode/code-runs/`에 남고, 생성 파일은 각 실행 폴더 안에 저장됨
- `최근 코딩 결과`는 대화별로 저장되어 같은 대화를 다시 열거나 새로고침한 뒤에도 복원됨
- Python, JS, Bash, C, C++, C#, Java, Kotlin, HTML, CSS를 처리

### 5.3 루틴 탭

- 루틴 생성/수정/즉시 실행/활성화/비활성화
- 브라우저 에이전트 모드와 일반 코드 실행 모드가 공존
- 브라우저 에이전트는 루틴별 도구 프로필을 선택할 수 있으며, 기본값은 `playwright_only`이고 `desktop_control`은 명시 opt-in이다
- `desktop_control`은 현재 macOS에서만 지원하며, Playwright 우선 + 필요 시 데스크톱 제어를 추가로 사용한다. 이 프로필에서는 로그인과 다운로드를 허용한다
- 브라우저 에이전트는 가능하면 `시작 URL`을 함께 주는 것이 가장 안정적이다. 대시보드에서는 `실행 모드 -> 브라우저 에이전트`를 고르고 `시작 URL`, `에이전트 모델`, `도구 프로필`을 함께 지정한다
- 루틴별 `텔레그램 봇 응답` on/off를 지원한다. 생성 시 기본값은 `켜기`이고, 상세 패널에서 나중에 바로 바꿀 수 있다
- `텔레그램 봇 응답`이 꺼져 있으면 예약 실행 결과는 자동 전송하지 않는다. 대신 `텔레그램 테스트`와 실행 이력 `재전송`은 명시적 전송이므로 계속 동작한다
- CLI/텔레그램에서는 예를 들어 `/routine create browser --model gpt-5.4 --url https://example.com --tool-profile desktop_control 로그인 후 CSV를 다운로드해서 요약해줘`처럼 생성할 수 있다
- 실행 산출물은 `workspace/coding/routines/` 아래에 저장되며, 브라우저 에이전트의 스크린샷과 다운로드 파일은 `workspace/coding/routines/<routine-id>/assets/<runTs>/` 아래에 실행별로 남는다
- 실행 이력의 `telegram=not_applicable`는 웹에서 수동 실행한 건이라 텔레그램 전송을 시도하지 않았다는 의미이며, 예약 실행(`scheduler`)이나 테스트 전송에서는 실제 전송 결과가 남음

### 5.4 로직 탭

- 로직 탭은 `logic.graph.v1` 그래프를 편집하고 실행하는 웹 전용 캔버스다
- 데스크톱에서는 좌측 `1/4` 그래프 목록, 중앙 `2/4` 캔버스, 우측 `1/4` 노드 팔레트, 하단 인스펙터로 배치된다
- 모바일 세로 레이아웃에서는 `목록 / 캔버스 / 팔레트 / 인스펙터` 섹션 탭으로 바뀐다
- 새 그래프를 만들면 기본으로 `시작` 노드 하나만 놓이고, 초기 보기 배율은 `100%` 기준이다
- 좌측 목록에서 그래프 이름을 바로 수정하고 저장/실행할 수 있다
- 좌측 작업 흐름 목록과 우측 노드 팔레트는 패널 안에서 드래그해 위아래로 스크롤할 수 있다
- 배경 드래그로 팬, 휠로 줌, 노드 카드 전체 드래그로 이동, 출력 포트 점에서 입력 포트 점으로 드래그해 연결한다
- 저장된 그래프를 다시 열었을 때 이전 뷰포트가 화면 밖으로 밀려난 경우에는 캔버스가 자동으로 맞춰져 노드가 바로 보인다
- 연결 중에는 프리뷰 선이 마우스를 따라오고, 노드는 최소 크기 아래로 줄어들지 않게 제한된다
- 노드 팔레트는 `흐름`, `AI`, `자동화`, `데이터/도구`, `운영` 그룹으로 고정돼 있다
- 하단 인스펙터에서 그래프 메타데이터, 선택 노드 설정, 실행 로그를 보고 `직접 입력 / 목록에서 고르기 / 연결해서 받기`로 값을 넣는다
- 파일/메모리 경로가 필요한 노드는 `찾아보기`로 경로를 고를 수 있다
- 저장 단위는 기존 루틴 저장소를 확장한 `logic_graph` 모드라서 `~/.omninode/routines.json`에 함께 저장되지만, 일반 루틴 탭 목록에는 나오지 않는다
- 런타임 산출물은 `workspace/.runtime/logic/<graph-id>/<run-id>/graph.json`, `snapshot.json`, `events.log`에 남는다
- 현재 로직 그래프 편집/실행 제어는 텔레그램 slash 명령이 아니라 대시보드 WebSocket 경로가 기준이다

### 5.5 설정 탭

- Telegram / Groq / Gemini / Cerebras / Codex 키 관리
- Copilot / Codex 상태 조회와 로그인 시작
- `프로젝트 문맥` 패널에서 AGENTS, fallback 문서, `.omni/skills`, `.omni/commands` 스캔
- `작업 계획` 패널에서 계획 생성, 리뷰, 승인, 실행
- `Background Task Graph` 패널에서 plan 기반 DAG 생성, 실행, task 로그 확인
- `Notebook / Handoff` 패널에서 learnings / decisions / verification / handoff 관리
- `환경 진단` 패널에서 최근 doctor 보고서 조회와 재실행
- `Routing Policy`와 provider/tool/rag 운영 패널 확인

## 6. 텔레그램 사용

텔레그램 자격정보가 있으면 미들웨어가 자동으로 롱폴링을 시작합니다.

핵심 변화:

- 슬래시(`/`) 없이도 텔레그램 자연어로 대부분의 제어가 가능함
- 예: `단일 모드로 바꿔`, `Codex로 바꿔`, `단일 코딩으로 로그인 페이지 만들어줘`, `단일 코딩 제공자를 Codex로 바꿔`, `다중 코딩 워커 Gemini 모델을 gemini-2.5-pro로 설정해`, `최근 코딩 결과 보여줘`, `리팩터 상태 보여줘`, `루틴 수정 rt-... 매일 9시에 서버 상태 점검`, `환경 진단해줘`, `계획 생성 서버 상태 점검 자동화`, `노트북 보여줘`, `메모리 초기화해줘`, `handoff 만들어줘`
- 자연어 제어는 `doctor`, `plan`, `task`, `notebook`, `handoff`, `memory`, `routine`, `coding`, `Safe Refactor`, `LLM 상태/모델 목록/사용량`까지 연결됨
- 일반 질문은 이제 불필요한 명령 해석 LLM 1회를 거치지 않고 바로 대화 경로로 들어가며, 명시적 제어 문장만 자연어 명령 해석을 탐
- 텔레그램 응답은 길이가 길더라도 중복 chunk 재전송을 피하고, 생성 중에는 typing/placeholder 메시지로 진행 상태를 먼저 보여줌
- 텔레그램 polling offset은 `~/.omninode/telegram_update_offset.txt`에 저장되고 단일 인스턴스 락 파일 `telegram_update_loop.lock`으로 중복 polling을 막음
- 텔레그램 API가 일시 실패해서 최종 응답 전송이 안 되면 `~/.omninode/telegram_reply_outbox.json`에 넣어 두고 다음 polling에서 자동 재전송을 시도함
- 정확한 제어가 필요할 때만 slash를 쓰면 됨
- 고급 Safe Refactor preview는 `refactor preview <path> <start> <end> ::: 새 코드`처럼 slash 없는 command-like 입력도 가능함

## 9. 자주 헷갈리는 점

- 루트 `package.json`에는 앱 실행용 `npm start`가 없습니다.
- 루트 `npm test`는 `node ./scripts/run-omninode-tests.mjs`를 실행합니다. `pnpm`은 필요 없습니다.
- 대시보드는 별도 프런트엔드 dev server 없이 미들웨어가 직접 서빙합니다.
- 검색 기능 품질은 일반 대화 품질과 다르게 grounding, guard, fallback composer, 캐시 스냅샷 준비 상태에 크게 좌우됩니다.
- 루틴 이력의 `telegram=not_applicable`는 실패가 아니라 수동 실행 source가 `web`이어서 텔레그램 전송을 건너뛴 경우입니다.
- 로직 그래프는 `~/.omninode/routines.json`에 같이 저장되지만 일반 루틴 탭 목록과 텔레그램 `/routine list`에는 섞이지 않습니다.
- `sqlite3`가 없으면 메모리 인덱스 초기화가 실패할 수 있습니다.
- 지워도 되는 것과 보존할 것은 [CLEANUP.md](CLEANUP.md)를 같이 보면 판단이 빨라집니다.
