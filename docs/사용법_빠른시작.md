# Omni-node 사용법 빠른 시작

업데이트 기준: 2026-03-08

이 문서는 저장소를 처음 다시 열었을 때 가장 빠르게 부팅하고, 대시보드에 접속하고, 핵심 기능을 확인하는 절차만 정리합니다.

## 1. 사전 준비

최소 권장 도구:

| 항목 | 용도 |
|---|---|
| `cc` | `omninode-core` 빌드 |
| `.NET SDK 9` | `omninode-middleware` 빌드/실행 |
| `python3` | 샌드박스 실행기, 코딩 워크플로 |
| `sqlite3` | 메모리 인덱스 초기화 |
| `node` + `npm` | Playwright 및 회귀 스크립트 |
| `gh` + `copilot` | Copilot CLI 연동 시 |
| `codex` | Codex CLI 연동 시 |

선택 항목:

- Telegram Bot Token / Chat ID
- Gemini / Groq / Cerebras API Key
- Codex API Key 또는 Codex CLI 로그인

## 2. 가장 빠른 부팅 순서

### 2.1 코어 실행

```bash
make -C apps/omninode-core
./apps/omninode-core/omninode_core
```

코어는 `/tmp/omninode_core.<uid>.sock` UDS 소켓과 `/tmp/omninode.<uid>.lock` 락 파일을 사용합니다.

### 2.2 미들웨어 실행

다른 터미널에서:

```bash
dotnet run --project apps/omninode-middleware/OmniNode.Middleware.csproj
```

기본 포트는 `8080`입니다.

### 2.3 대시보드 접속

- 대시보드: `http://127.0.0.1:8080/`
- 헬스체크: `http://127.0.0.1:8080/healthz`
- 준비상태: `http://127.0.0.1:8080/readyz`

## 3. 인증 방식

대시보드는 WebSocket OTP 인증을 사용합니다.

기본 흐름:

1. 브라우저에서 대시보드 접속
2. `OTP 요청` 실행
3. 텔레그램이 설정되어 있으면 텔레그램으로 OTP 수신
4. 텔레그램이 없으면 콘솔에 로컬 fallback OTP가 출력됨
5. OTP 입력 후 세션 인증 완료

기본적으로 `OMNINODE_ENABLE_LOCAL_OTP_FALLBACK=true`라서 텔레그램 미설정 상태에서도 로컬 테스트가 가능합니다.

## 4. 추천 환경변수 예시

시크릿은 직접 환경변수보다 `*_FILE` 방식이 안전합니다. 파일 권한은 `0600` 이상으로 맞추는 것이 전제입니다.

```bash
mkdir -p "$HOME/.omni/keys"

printf 'YOUR_GEMINI_KEY' > "$HOME/.omni/keys/gemini_api_key"
printf 'YOUR_GROQ_KEY' > "$HOME/.omni/keys/groq_api_key"
printf 'YOUR_CEREBRAS_KEY' > "$HOME/.omni/keys/cerebras_api_key"
chmod 600 "$HOME/.omni/keys/"*

export OMNINODE_GEMINI_API_KEY_FILE="$HOME/.omni/keys/gemini_api_key"
export OMNINODE_GROQ_API_KEY_FILE="$HOME/.omni/keys/groq_api_key"
export OMNINODE_CEREBRAS_API_KEY_FILE="$HOME/.omni/keys/cerebras_api_key"
export OMNINODE_WORKSPACE_ROOT="/Users/songhabin/Omni-node/workspace/coding"
export OMNINODE_GATEWAY_STARTUP_PROBE=1
```

이 문서의 워크스페이스 예시는 항상 canonical 경로인 `/Users/songhabin/Omni-node/workspace/coding`을 기준으로 합니다. 루트 `coding`은 하위 호환 alias입니다.

Codex를 API Key로 쓸 경우:

```bash
printf 'YOUR_CODEX_KEY' > "$HOME/.omni/keys/codex_api_key"
chmod 600 "$HOME/.omni/keys/codex_api_key"
export OMNINODE_CODEX_API_KEY_FILE="$HOME/.omni/keys/codex_api_key"
```

텔레그램을 함께 붙일 경우:

```bash
printf 'YOUR_TELEGRAM_BOT_TOKEN' > "$HOME/.omni/keys/telegram_bot_token"
printf 'YOUR_TELEGRAM_CHAT_ID' > "$HOME/.omni/keys/telegram_chat_id"
chmod 600 "$HOME/.omni/keys/telegram_"*

export OMNINODE_TELEGRAM_BOT_TOKEN_FILE="$HOME/.omni/keys/telegram_bot_token"
export OMNINODE_TELEGRAM_CHAT_ID_FILE="$HOME/.omni/keys/telegram_chat_id"
export OMNINODE_TELEGRAM_ALLOWED_USER_ID="YOUR_TELEGRAM_USER_ID"
```

## 5. 대시보드 사용 순서

### 5.1 대화 탭

- 단일 / 오케스트레이션 / 다중 대화 모드 지원
- 메모리 노트 연결, URL 컨텍스트, 웹 검색 on/off 사용 가능
- 최신정보 질의는 Gemini grounding 우선 경로와 `SearchAnswerGuard`, evidence/cache fallback 정책 영향을 받음

### 5.2 코딩 탭

- 단일 / 오케스트레이션 / 다중 코딩 모드 지원
- 실행 결과는 기본적으로 `~/.omninode/code-runs/`에 쌓임
- Python, JS, Bash, C, C++, C#, Java, Kotlin, HTML, CSS를 처리

### 5.3 루틴 탭

- 루틴 생성/수정/즉시 실행/활성화/비활성화
- 브라우저 에이전트 모드와 일반 코드 실행 모드가 공존
- 실행 산출물은 `workspace/coding/routines/` 아래에 저장됨

### 5.4 설정 탭

- Telegram / Groq / Gemini / Cerebras / Codex 키 관리
- Copilot / Codex 상태 조회와 로그인 시작
- provider/tool/rag 운영 패널 확인

## 6. 텔레그램 사용

텔레그램 자격정보가 있으면 미들웨어가 자동으로 롱폴링을 시작합니다.

대표 명령:

- `/llm status`
- `/memory clear`
- `/routine list`
- `/routine create <요청>`
- `/routine run <routine-id>`

텔레그램이 미설정이면 폴링은 사실상 비활성 상태로 대기하고, OTP 전송도 건너뜁니다.

## 8. 종료

- 코어와 미들웨어는 각각 실행 중인 터미널에서 `Ctrl+C`
- launchd/systemd 상시 운영은 `deploy/` 템플릿을 참고

## 9. 자주 헷갈리는 점

- 루트 `package.json`에는 앱 실행용 `npm start`가 없습니다.
- 루트 `npm test`는 대시보드 구문 검사와 미들웨어 빌드를 함께 수행하는 통합 검증 엔트리입니다.
- 대시보드는 별도 프런트엔드 dev server 없이 미들웨어가 직접 서빙합니다.
- 검색 기능 품질은 일반 대화 품질과 다르게 grounding, guard, fallback composer, 캐시 스냅샷 준비 상태에 크게 좌우됩니다.
- `sqlite3`가 없으면 메모리 인덱스 초기화가 실패할 수 있습니다.
- 지워도 되는 것과 보존할 것은 [CLEANUP.md](CLEANUP.md)를 같이 보면 판단이 빨라집니다.
