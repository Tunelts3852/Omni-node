# Omni-node 검증 가이드

업데이트 기준: 2026-03-08

이 문서는 저장소에서 바로 돌려볼 수 있는 빌드/스모크/회귀 검증 명령을 모아 둔 운영용 체크리스트입니다.

기억할 순서는 `부팅 검증 -> 기본 건강검진 -> 실행기 검증` 세 단계입니다.

## 1. 이번 문서 정리 시 실제로 확인한 항목

아래 명령은 2026-03-08 현재 로컬 워크스페이스에서 직접 확인했습니다.

| 명령 | 결과 |
|---|---|
| `make -C apps/omninode-core` | 성공 |
| `python3 apps/omninode-sandbox/executor.py --code "print('ok')"` | 성공, `ok` 출력 |
| `npm test` | 성공 (`test:dashboard` + `test:middleware`) |

## 2. 대표 3단 검증

### 2.1 부팅 검증

```bash
make -C apps/omninode-core
dotnet run --project apps/omninode-middleware/OmniNode.Middleware.csproj
```

브라우저 또는 `curl`로 아래를 확인합니다.

- `http://127.0.0.1:8080/`
- `http://127.0.0.1:8080/healthz`

### 2.2 기본 건강검진

```bash
npm test
```

### 2.3 실행기 검증

```bash
python3 apps/omninode-sandbox/executor.py --code "print('ok')"
```

### 2.4 필요 시 세부 빌드/구문 점검

```bash
dotnet build apps/omninode-middleware/OmniNode.Middleware.csproj
node --check apps/omninode-dashboard/app.js
node --check apps/omninode-dashboard/worker.js
```

## 3. Playwright/Node 기반 회귀

루트 `package.json`의 `npm test`는 현재 대시보드 구문 검사와 미들웨어 빌드를 묶는 기본 검증 엔트리입니다. 그 외에 브라우저/회귀 스크립트는 필요 시 개별 실행합니다.

먼저 의존성 준비:

```bash
npm ci
npx playwright install --with-deps chromium
```

대표 스크립트:

| 명령 | 의미 |
|---|---|
| `node apps/omninode-dashboard/check-guard-threshold-lock.js` | guard threshold 잠금 검증 |
| `node apps/omninode-dashboard/check-chat-multi-utils.js` | chat multi 유틸 계약 검증 |
| `node apps/omninode-dashboard/check-guard-retry-timeline-browser-e2e.js` | retry timeline 브라우저 E2E |
| `node apps/omninode-dashboard/check-ops-flow-performance.js` | 운영 패널 성능 점검 |
| `node apps/omninode-middleware/check-p3-guard-smoke.js` | guard smoke |
| `node apps/omninode-middleware/check-startup-probe-state.js` | startup probe 상태 검증 |
| `node apps/omninode-middleware/check-guard-alert-dispatch.js` | guard alert dispatch 회귀 |

## 4. GitHub Actions에 반영된 대표 회귀

현재 루트 `.github/workflows/`에는 아래 워크플로가 있습니다.

| 워크플로 | 의미 |
|---|---|
| `guard-alert-dispatch-regression.yml` | dispatch mock/live 회귀 |
| `guard-retry-timeline-browser-e2e-regression.yml` | retry timeline 브라우저 E2E + p3/p7 묶음 회귀 |

공통 특징:

- Node.js 22 사용
- .NET 9 사용
- Gemini 키를 secure file 방식으로 강제
- 실행 결과를 artifact로 업로드

## 5. 수동 실행 시 유용한 순서

권장 순서:

1. `make -C apps/omninode-core`
2. `dotnet run --project apps/omninode-middleware/OmniNode.Middleware.csproj`
3. 대시보드 또는 `healthz` 확인
4. `npm test`
5. `python3 apps/omninode-sandbox/executor.py --code "print('ok')"`
6. 필요하면 `dotnet build apps/omninode-middleware/OmniNode.Middleware.csproj`
7. 필요한 경우 Playwright/회귀 스크립트 실행

## 6. 라이브 회귀 전에 필요한 것

아래 조건이 없으면 일부 회귀는 실패하거나 의미가 줄어듭니다.

- Gemini API Key
- Chromium 설치
- `.NET 9 SDK`
- `sqlite3`
- 텔레그램 또는 외부 webhook 대상이 필요한 테스트의 경우 해당 자격정보

## 7. 헷갈리기 쉬운 점

- `npm test`는 현재 대시보드 구문 검사와 미들웨어 빌드를 함께 묶는 기본 검증 명령입니다.
- 대시보드 앱은 브라우저 전역 객체를 쓰므로 실행 테스트와 단순 구문 점검을 구분해서 봐야 합니다.
- 검색 관련 라이브 회귀는 생성기 자체보다 grounding, guard 상태 파일, fallback composer, 캐시/아티팩트 준비 상태 영향이 큽니다.
- 루틴 검증은 서버가 실제로 떠 있는 상태에서 보는 편이 훨씬 정확합니다.
