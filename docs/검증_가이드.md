# Omni-node 검증 가이드

업데이트 기준: 2026-03-10

이 문서는 저장소에서 바로 돌려볼 수 있는 빌드/스모크/회귀 검증 명령을 모아 둔 운영용 체크리스트입니다.

기억할 순서는 `부팅 검증 -> 기본 건강검진 -> doctor 진단 -> 실행기 검증` 네 단계입니다.

아래 명령은 2026-03-10 현재 로컬 워크스페이스 기준으로 다시 확인하거나 현재 기본 검증 경로로 유지하고 있습니다.

## 2. 대표 4단 검증

### 2.1 부팅 검증

```bash
make -C apps/omninode-core
dotnet run --project apps/omninode-middleware/OmniNode.Middleware.csproj
```

브라우저 또는 `curl`로 아래를 확인합니다.

- `http://127.0.0.1:8080/`
- `http://127.0.0.1:8080/healthz`
- `http://127.0.0.1:8080/readyz`
- `http://127.0.0.1:8080/health`
- `http://127.0.0.1:8080/ready`

`readyz`는 WebSocket roundtrip이 1회 관측돼야 200이 됩니다. 현재 대시보드는 연결 직후 초기 `ping`을 자동으로 보내므로, 페이지가 정상 연결되면 별도 수동 ping 없이도 준비상태가 올라가야 합니다.

미들웨어 단독 실행 검증 시에는 코어를 수동으로 먼저 띄우는 것이 가장 명확하지만, 현재는 `core_socket`이 죽어 있으면 미들웨어가 `omninode_core` 자동 부트스트랩도 시도합니다.

### 2.2 기본 건강검진

```bash
npm test
```

현재 루트 `npm test`는 `scripts/run-omninode-tests.mjs`를 실행하며 아래를 순서대로 확인합니다.

- `node scripts/check-repo-hygiene.mjs`
- `apps/omninode-dashboard/app.js`와 `modules/**/*.{js,mjs}`에 대한 `node --check`
- `node apps/omninode-dashboard/check-chat-multi-utils.js`
- `node apps/omninode-dashboard/check-ops-flow-performance.js --events 256 --iterations 20`
- `node apps/omninode-dashboard/check-dashboard-server-message-router.mjs`
- `dotnet build apps/omninode-middleware/OmniNode.Middleware.csproj`
- `python3 apps/omninode-sandbox/executor.py --code "print('ok')"`

### 2.3 doctor 진단

```bash

```

`doctor`는 부팅 검증과 별개의 환경 진단 단계다. 서버가 떠 있는지 확인하는 절차와 의존성/상태 구성을 점검하는 절차를 분리해서 본다.

### 2.4 실행기 검증

```bash

```

### 2.6 Safe Refactor smoke

실제 refactor core 검증은 임시 워크스페이스에서 아래 순서로 확인한다.

1. 임시 파일 생성
2. anchor read
3. preview 생성
4. 외부 수정으로 stale mismatch 재현
5. 새 preview 생성 후 apply
6. `lsp_rename -> refactor_apply`로 구조적 rename 적용
7. `ast_replace -> refactor_apply`로 pattern rewrite 적용

기대 결과 예시:

- `previewSafe=True`
- `staleIssueCount=1`
- `applySafe=True`
- `finalLine2=beta-approved`
- `lspRenameApplied=True`
- `astReplaceApplied=True`

### 2.7 최근 코딩 결과 복원 smoke

코딩 탭에서 아래 순서로 확인한다.

1. 같은 대화에서 코딩을 1회 실행
2. `최근 코딩 결과` 도크가 뜨는지 확인
3. 페이지 새로고침
4. 같은 대화를 다시 선택
5. `최근 코딩 결과` 도크와 오버레이 내용이 복원되는지 확인

### 2.8 최근 코딩 결과 실행/프리뷰 smoke

같은 코딩 대화에서 아래를 확인한다.

1. `최근 코딩 결과` 오버레이 열기
2. 첫 파일 칩 클릭 후 프리뷰가 실제 파일 내용으로 뜨는지 확인
3. `실행` 버튼 클릭
4. 엔트리 실행 명령이 표시되는지 확인
5. `stdout`, `stderr`, 작업 폴더, 엔트리 파일이 분리된 카드로 보이는지 확인
6. HTML이면 iframe 프리뷰, Python CLI면 실제 stdout/stderr, 입력형 CLI면 `stdin 입력` 후 재실행이 되는지 확인

### 2.9 코딩 제공자 매트릭스 smoke

코딩탭 provider/model/language/mode 매트릭스는 아래 스크립트로 다시 돌립니다.

표준 제공자 전체 매트릭스:

```bash
node apps/omninode-middleware/check-coding-provider-smoke.js \
  --providers codex,copilot,gemini \
  --profile complex \
  --languages python,javascript,java,c,html \
  --modes single,orchestration,multi \
  --skip-build \
  --timeout-ms 300000 \
  --write workspace/.runtime/coding-mode-matrix-standard.json
```

lean 제공자 전체 매트릭스:

```bash
node apps/omninode-middleware/check-coding-provider-smoke.js \
  --providers groq,cerebras \
  --profile complex \
  --languages python,javascript,java,c,html \
  --modes single,orchestration,multi \
  --skip-build \
  --timeout-ms 300000 \
  --write workspace/.runtime/coding-mode-matrix-lean.json
```

현재 문서 기준으로 직접 확인한 검증 규칙:

- `stdout` exact match
- 생성/수정 파일 존재 확인
- 파일 내용 확인
- `java`, `c`는 실제 컴파일 후 실행
- `html`은 Playwright로 visible text, selector 수, style 값 확인
- `python` GUI/게임 과제는 최종 검증에서 `python main.py`를 그대로 돌리지 않고, 변경된 `.py` 전체에 대해 `python3 -m py_compile ...`로 bounded 문법 검증하며, 명시적으로 허용하지 않은 `pygame` 같은 서드파티 의존성은 검증 단계에서 실패 처리 후 표준 라이브러리 재작성 복구를 유도

## 3. Playwright/Node 기반 회귀

루트 `npm test`는 로컬 smoke runner를 사용하고, 그 외의 브라우저/회귀 스크립트는 필요 시 개별 실행합니다.

먼저 의존성 준비:

```bash
npm ci
npx playwright install --with-deps chromium
```

대표 스크립트:

| 명령 | 의미 |
|---|---|
| `node apps/omninode-dashboard/check-guard-threshold-lock.js` | guard threshold 잠금 검증 |
| `node apps/omninode-dashboard/check-chat-multi-utils.js` | chat multi 유틸 계약 검증 |
| `node apps/omninode-dashboard/check-guard-retry-timeline-browser-e2e.js` | retry timeline 브라우저 E2E |
| `node apps/omninode-dashboard/check-ops-flow-performance.js` | 운영 패널 성능 점검 |
| `node apps/omninode-middleware/check-p3-guard-smoke.js` | guard smoke |
| `node apps/omninode-middleware/check-startup-probe-state.js` | startup probe 상태 검증 |
| `node apps/omninode-middleware/check-guard-alert-dispatch.js` | guard alert dispatch 회귀 |
| `node apps/omninode-middleware/check-coding-provider-smoke.js ...` | 코딩 provider/model/language/mode 매트릭스 smoke |

## 5. 수동 실행 시 유용한 순서

권장 순서:

1. `make -C apps/omninode-core`
2. `dotnet run --project apps/omninode-middleware/OmniNode.Middleware.csproj`
3. 대시보드 또는 `healthz` 확인
4. `npm test`
5. `dotnet run --project apps/omninode-middleware/OmniNode.Middleware.csproj -- doctor --json`
6. `python3 apps/omninode-sandbox/executor.py --code "print('ok')"`
7. 필요하면 `dotnet build apps/omninode-middleware/OmniNode.Middleware.csproj`
8. 코딩 라우팅이나 언어별 동작을 만졌다면 `check-coding-provider-smoke.js`로 관련 provider/language/mode만 재검증
9. 계획 기능을 만졌다면 설정 탭 `작업 계획` 패널 또는 `/plan ...` 슬래시 명령으로 생성/리뷰/승인/실행 확인
10. Safe Refactor를 만졌다면 코딩 탭 `Safe Refactor` 도크/오버레이 또는 임시 smoke로 stale apply 차단 확인
11. 코딩 결과 복원을 만졌다면 새로고침 후 `최근 코딩 결과` 복원 smoke 확인
12. 필요한 경우 Playwright/회귀 스크립트 실행

## 6. 라이브 회귀 전에 필요한 것

아래 조건이 없으면 일부 회귀는 실패하거나 의미가 줄어듭니다.

- Gemini API Key
- Chromium 설치
- `.NET 9 SDK`
- `sqlite3`
- 텔레그램 또는 외부 webhook 대상이 필요한 테스트의 경우 해당 자격정보

## 7. 헷갈리기 쉬운 점

- `npm test`는 루트 smoke runner 기준이라 `pnpm` 없이도 실행됩니다.
- 대시보드 앱은 브라우저 전역 객체를 쓰므로 실행 테스트와 단순 구문 점검을 구분해서 봐야 합니다.
- 검색 관련 라이브 회귀는 생성기 자체보다 grounding, guard 상태 파일, fallback composer, 캐시/아티팩트 준비 상태 영향이 큽니다.
- 루틴 검증은 서버가 실제로 떠 있는 상태에서 보는 편이 훨씬 정확합니다.
