# Omni-node 환경변수 및 상태 파일

업데이트 기준: 2026-03-10

이 문서는 경로, 상태, 시크릿을 다시 확인할 때 보는 기준 문서입니다. 전체 환경변수 목록과 기본값 정의는 `apps/omninode-middleware/src/AppConfig.cs`가 가장 정확합니다.

## 1. 기준 경로와 기억할 원칙

기본으로 기억할 canonical 경로는 아래와 같습니다.

| 구분 | 기준 경로 | 메모 |
|---|---|---|
| 앱 소스 | `apps/` | 실제 런타임 소스 |
| 문서 | `docs/` | canonical 문서 위치 |
| 작업공간 | `/path/to/Omni-node/workspace/coding` | `OMNINODE_WORKSPACE_ROOT` 기본 예시 |

원칙:

- 문서와 쉘 export 예시에서는 canonical 경로만 기본값으로 표기합니다.
- 루트 `coding`, `runtime`, `omninode-*`, 루트 문서 파일은 하위 호환 alias로만 봅니다.
- 실제 운영에서 `OMNINODE_WORKSPACE_ROOT`를 명시할 때도 canonical 경로를 우선 사용합니다.

## 2. 시크릿 로딩 우선순위

LLM/텔레그램 시크릿은 아래 순서로 로드됩니다.

1. macOS Keychain
2. 로컬 secure store
3. `*_FILE`
4. 직접 환경변수

중요 사항:

- 직접 환경변수는 deprecated 경고를 남깁니다.
- `*_FILE`은 권한이 `0600` 또는 그보다 엄격해야 합니다.
- 문서의 `*_FILE` 예시는 `~/.omninode/keys/` 디렉터리를 기준으로 듭니다.
- 로컬 secure store 기본 경로는 `~/.config/omninode/secrets.json`입니다.

## 4. 기본 상태 파일 위치

상태 루트는 기본적으로 `~/.omninode`입니다. 홈 디렉터리를 얻지 못하면 `/tmp`로 fallback 합니다.

상태는 아래 세 층으로 기억하면 덜 헷갈립니다.

| 층 | 대표 경로 | 의미 |
|---|---|---|
| 영속 상태 | `~/.omninode` | 설정, 세션, 대화, 루틴 정의 원본 |
| 작업 산출물 | `workspace/` | 루틴 결과, 회귀 아티팩트, 작업 대상 파일 |
| 임시 실행 상태 | `/tmp`, `workspace/.runtime`, `workspace/runtime` | 소켓, 락, 임시 스냅샷 |

### 4.1 `~/.omninode` 아래 주요 파일

| 경로 | 의미 |
|---|---|
| `llm_usage.json` | LLM 사용량 상태 |
| `copilot_usage.json` | Copilot 사용량 상태 |
| `conversations.json` | 대화 저장소(메시지, linked memory, 최근 코딩 결과 snapshot 및 다중 코딩 비교 메타 포함) |
| `auth_sessions.json` | 인증 세션 저장소 |
| `guard_retry_timeline.json` | guard retry timeline 상태 |
| `gateway_health.json` | gateway health 상태 |
| `gateway_startup_probe.json` | startup probe 상태 |
| `telegram_update_offset.txt` | 텔레그램 polling next update checkpoint |
| `telegram_update_loop.lock` | 텔레그램 polling 단일 인스턴스 락 파일 |
| `routines.json` | 루틴 정의 원본 |
| `routing-policy.json` | category별 provider override 정책 |
| `plans/index.json` | 계획 인덱스 |
| `plans/<plan-id>/plan.json` | 계획 본문 |
| `plans/<plan-id>/review.json` | 리뷰 결과 |
| `plans/<plan-id>/execution.json` | 최근 실행 결과 |
| `tasks/index.json` | task graph 인덱스 |
| `tasks/<graph-id>.json` | task graph 스냅샷 |
| `notebooks/<project-key>/learnings.md` | 프로젝트 learnings 기록 |
| `notebooks/<project-key>/decisions.md` | 프로젝트 decisions 기록 |
| `notebooks/<project-key>/verification.md` | 프로젝트 verification 기록 |
| `notebooks/<project-key>/handoff.md` | 최신 handoff 문서 |

| `commands/` | 전역 command template 루트 |

| `code-runs/` | 코드 실행 결과 메타데이터/로그 이력 |
| `memory-index/main.sqlite` | 메모리 검색 인덱스 DB |

### 4.2 `workspace/` 아래 주요 경로

| 경로 | 의미 |
|---|---|
| `workspace/coding/` | 앱이 다루는 작업 대상 워크스페이스 |
| `workspace/coding/runs/` | 코딩 탭 세 모드가 공유하는 실행별 작업 폴더 |
| `workspace/coding/routines/` | 루틴 실행 결과 이력 |
| `workspace/.runtime/` | 회귀 결과/임시 분석물 |
| `workspace/.runtime/coding-*.json` | provider/model/language smoke 결과 리포트 |
| `workspace/.runtime/tasks/` | task graph 실행 로그와 result.json |
| `workspace/.runtime/refactor-preview/` | Safe Refactor preview JSON |
| `workspace/runtime/` | 현재 세션 상태 스냅샷 |
| `workspace/coding/.omni/skills/` | 프로젝트 전용 skill manifest |
| `workspace/coding/.omni/commands/` | 프로젝트 전용 command template |

루틴은 특히 아래처럼 구분해서 봅니다.

- 루틴 정의를 보존하려면 `~/.omninode/routines.json`이 중요합니다.
- provider chain override를 유지하려면 `~/.omninode/routing-policy.json`을 같이 봐야 합니다.
- 코딩 탭 생성 파일까지 보존하려면 `workspace/coding/runs/`를 같이 봐야 합니다.
- 다중 코딩의 모델별 실제 산출물까지 보존하려면 `workspace/coding/runs/<실행별 폴더>/<provider>-<model>/` 하위 폴더까지 같이 봐야 합니다.
- provider/model/language 회귀 결과까지 보존하려면 `workspace/.runtime/coding-*.json`을 같이 봐야 합니다.
- 최근 코딩 결과 복원 상태까지 보존하려면 `~/.omninode/conversations.json`을 같이 봐야 합니다.
- 텔레그램 코딩의 최근 결과/파일 프리뷰도 같은 `conversations.json` snapshot과 `workspace/coding/runs/` 산출물을 같이 사용합니다.
- 최근 코딩 결과 `실행`은 검증 커맨드가 아니라 엔트리 실행 명령을 우선 구성하며, GUI/인터랙티브 앱은 별도 실행 유지 경로를 사용할 수 있습니다.
- 과거 실행 산출물까지 보존하려면 `workspace/coding/routines/`도 같이 봐야 합니다.
- Task graph 상태를 복구하려면 `~/.omninode/tasks/`와 `workspace/.runtime/tasks/`를 같이 봐야 합니다.
- notebook / handoff를 이어가려면 `~/.omninode/notebooks/`를 같이 봐야 합니다.
- Safe Refactor preview를 추적하려면 `workspace/.runtime/refactor-preview/`를 봅니다.
- 텔레그램 Safe Refactor의 `/refactor preview`도 같은 `workspace/.runtime/refactor-preview/` JSON을 사용합니다.
- project context를 추적하려면 루트 `AGENTS.md`, 필요한 경우 `AGENTS.override.md`, 그리고 `.omni/skills/`, `.omni/commands/`를 같이 봅니다.
- 루틴 이력에서 `telegramStatus=not_applicable`는 웹 수동 실행이라 텔레그램 전송 대상이 아니었다는 의미입니다.

### 4.3 `/tmp` 아래 주요 파일

코어와 일부 기본 경로는 `/tmp`를 직접 사용합니다.

| 경로 패턴 | 의미 |
|---|---|
| `/tmp/omninode_core.<uid>.sock` | 코어 UDS 소켓 |
| `/tmp/omninode.<uid>.lock` | 코어 단일 인스턴스 락 |
| `/tmp/omninode_audit.log` | 감사 로그 기본 위치 |

## 5. 운영에 자주 필요한 조합

### 5.1 최소 웹 대시보드 실행

```bash
export OMNINODE_WORKSPACE_ROOT="/path/to/Omni-node/workspace/coding"
export OMNINODE_ENABLE_LOCAL_OTP_FALLBACK=1
```

### 5.2 검색 기능 포함 실행

```bash
export OMNINODE_GEMINI_API_KEY_FILE="$HOME/.omninode/keys/gemini_api_key"
export OMNINODE_GEMINI_SEARCH_MODEL="gemini-3.1-flash-lite-preview"
```

### 5.3 텔레그램 포함 실행

```bash
export OMNINODE_TELEGRAM_BOT_TOKEN_FILE="$HOME/.omninode/keys/telegram_bot_token"
export OMNINODE_TELEGRAM_CHAT_ID_FILE="$HOME/.omninode/keys/telegram_chat_id"
export OMNINODE_TELEGRAM_ALLOWED_USER_ID="123456789"
```

## 6. 알아두면 좋은 구현 포인트

- 대시보드 경로는 실행 경로 기준으로 자동 탐색합니다.
- 워크스페이스 루트도 자동 탐색하지만, 실제 운영에서는 `OMNINODE_WORKSPACE_ROOT="/path/to/Omni-node/workspace/coding"`처럼 canonical 경로를 명시하는 편이 안전합니다.
- Codex는 API Key가 없더라도 CLI 로그인 상태가 살아 있으면 사용 가능할 수 있습니다.
- 시크릿 secure file은 권한이 느슨하면 읽지 않습니다.

## 7. 청소 전 판단 기준

- 재생성 가능한 것은 과감히 청소 가능
- 상태 원본과 실행 이력은 청소 전에 보존 여부를 먼저 판단

정리 기준표는 [CLEANUP.md](CLEANUP.md)를 함께 보면 됩니다.
